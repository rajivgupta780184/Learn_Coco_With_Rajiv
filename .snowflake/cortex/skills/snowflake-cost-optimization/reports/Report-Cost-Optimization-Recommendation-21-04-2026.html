P2 — MEDIUM   R3: Create resource monitors for spend governance

Finding: F-C2GovernanceComplexity: Low

Problem: No resource monitors exist. If a warehouse or service runs away, there is no automated guardrail.

Remediation: Create an account-level resource monitor with alert thresholds:

CREATE RESOURCE MONITOR account_spend_monitor
  WITH CREDIT_QUOTA = 50
  FREQUENCY = MONTHLY
  START_TIMESTAMP = IMMEDIATELY
  TRIGGERS
    ON 75 PERCENT DO NOTIFY
    ON 90 PERCENT DO NOTIFY
    ON 100 PERCENT DO SUSPEND;

ALTER ACCOUNT SET RESOURCE_MONITOR = account_spend_monitor;

Optionally create a warehouse-level monitor for COMPUTE_WH:

CREATE RESOURCE MONITOR compute_wh_monitor
  WITH CREDIT_QUOTA = 10
  FREQUENCY = MONTHLY
  START_TIMESTAMP = IMMEDIATELY
  TRIGGERS
    ON 80 PERCENT DO NOTIFY
    ON 100 PERCENT DO SUSPEND;

ALTER WAREHOUSE COMPUTE_WH SET RESOURCE_MONITOR = compute_wh_monitor;

Value: Prevents unexpected cost spikes. Pure governance — no direct credit savings.

Risk: Setting SUSPEND triggers will halt queries when the limit is reached. Ensure the quota is appropriate for expected workloads.

P2 — MEDIUM   R4: Monitor Cortex Code usage growth

Finding: F-AI2AI / Cortex CodeComplexity: Low

Problem: Cortex Code (Snowsight + CLI) accounts for 5.36 credits ($16.09) = 28.9% of total spend. This is productive developer tooling, but costs will scale with team size.

Remediation: Set up a recurring monitoring query to track growth:

SELECT
    DATE_TRUNC('week', start_time) AS week,
    service_type,
    ROUND(SUM(credits_used), 2) AS weekly_credits
FROM SNOWFLAKE.ACCOUNT_USAGE.METERING_HISTORY
WHERE service_type IN ('CORTEX_CODE_CLI', 'CORTEX_CODE_SNOWSIGHT')
  AND start_time >= DATEADD(day, -90, CURRENT_TIMESTAMP())
GROUP BY week, service_type
ORDER BY week DESC;

Value: Early visibility into cost trends as more developers adopt Cortex Code.

P3 — LOW   R5: Track query spillage trend on COMPUTE_WH

Finding: F-C3ComputeComplexity: Low

Problem: 8 queries spilled to local storage (avg ~98 MB). Currently benign for X-Small, but if query complexity grows, remote spillage may occur (performance + cost hit).

Remediation: Periodically check for remote spillage:

SELECT warehouse_name, COUNT(*) AS spill_count,
  ROUND(AVG(bytes_spilled_to_remote_storage),0) AS avg_remote,
  ROUND(AVG(bytes_spilled_to_local_storage),0) AS avg_local
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE start_time >= DATEADD(day, -7, CURRENT_TIMESTAMP())
  AND bytes_spilled_to_remote_storage > 0
GROUP BY warehouse_name;

If remote spillage becomes frequent, consider upsizing to Small:

ALTER WAREHOUSE COMPUTE_WH SET WAREHOUSE_SIZE = 'SMALL';

Value: Preventive — avoid future performance degradation and unexpected credit burn.
